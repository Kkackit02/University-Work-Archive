### **1. 힌트 1: 전체적인 컴파일 과정 (가장 중요!)**

시험에서 전체 흐름을 묻는 것은 거의 확실해 보입니다. 6단계의 **순서, 각 단계의 역할, 입력, 출력**을 반드시 암기해야 합니다.

| 단계 | 한글명 | 역할 및 핵심 키워드 | 입력 | 출력 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | 어휘 분석 | 소스 코드를 토큰(Token)으로 분리 (도구: Flex, 규칙 파일: `lexer.l`) | 소스 코드 문자열 | 토큰 스트림 |
| 2 | 구문 분석 | 토큰이 문법에 맞는지 검사하고 AST(추상 구문 트리) 생성 (도구: Bison, 규칙 파일: `parser.y`) | 토큰 스트림 | AST |
| 3 | 의미 분석 | 타입 검사, 변수 선언 여부 등 의미적 오류 검출 (자료구조: 심볼 테이블) | AST | (의미가 검증된) AST |
| 4 | 중간 코드 생성 | AST를 기계 독립적인 코드로 변환 (형식: 3-주소 코드) | AST | IR (중간 코드) |
| 5 | 코드 최적화 | 코드의 효율성 증대 (예: 상수 폴딩, 불필요한 코드 제거) | IR | 최적화된 IR |
| 6 | 코드 생성 | 실제 기계(x86)가 실행할 코드로 변환 (관리 대상: 스택 프레임, 레지스터) | (최적화된) IR | 타겟 기계 코드 |

### **2. 힌트 2: `parser.y` 파일 1줄 수정 (★ 출제 유력)**

#### **가능성 1 (가장 유력): 연산자 우선순위 변경**

*   내용: `%left`, `%right` 선언문의 순서를 바꿔 연산자 우선순위를 변경하는 것입니다.
*   핵심: `parser.y` 파일에서 `%left`, `%right` 키워드는 **나중에 선언될수록 우선순위가 높습니다.**
*   예시:
    ```bison
    // 수정 전: 곱셈 우선순위가 높음
    %left '+' '-'
    %left '*' '/'
    ```
    `a + b * c` → `a + (b * c)`
*   1줄 수정 (순서 변경):
    ```bison
    // 수정 후: 덧셈 우선순위가 높아짐
    %left '*' '/'
    %left '+' '-'
    ```
    `a + b * c` → `(a + b) * c`

---

#### **힌트 2의 다른 현실적 가능성 (심화)**

우선순위 변경이 가장 유력하지만, 아래의 시나리오들도 "1줄 수정"의 범주에 포함될 수 있습니다.

| 가능성 | 내용 | 예시 (1줄 수정) | 영향 및 시험 예측 | 현실성 |
| :--- | :--- | :--- | :--- | :--- |
| **2. 결합 법칙 변경** | 연산자의 계산 방향을 바꿉니다. | `%left '-'` → `%right '-'` | `10-5-3`이 `(10-5)-3=2`에서 `10-(5-3)=8`로 바뀜. "수정 후 결과값은?" 유형의 문제 가능. | **높음** |
| **3. 의미 액션 수정** | 문법은 같지만 실제 동작을 바꿉니다. | `... { $$ = new_binop_expr(BIN_ADD, ...); }` → `... { $$ = new_binop_expr(BIN_SUB, ...); }` | `+` 기호가 뺄셈으로 동작. "수정 후 `a+b`의 결과는?" (정답: a-b) 유형의 문제 가능. | **높음** |
| **4. 새 규칙 추가** | 언어의 문법을 간단하게 확장합니다. | `stmt: ... \| ';' { $$ = NULL; };` | 기존에 오류였던 ` ; ` (빈 문장)을 허용하게 됨. "이 수정으로 가능해진 문법은?" 유형의 문제 가능. | **중간** |

### **3. 힌트 3: 3-주소 형태 (Intermediate Representation)**

*   개념: 중간 코드(IR)의 한 종류로, `결과 = 피연산자1 연산자 피연산자2` 형태의 단순한 명령어를 말합니다. (`result = op1 op op2`)
*   핵심: 복잡한 표현식을 여러 개의 간단한 3-주소 코드로 나누는 과정을 이해해야 합니다. 이 때, 중간 계산 결과를 저장하기 위해 임시 변수(temporary variable, 예: `t0`, `t1`)가 사용됩니다.
*   예시: `x = a + b * 5` 를 3-주소 코드로 변환하는 과정
    1.  `t0 = b * 5`
    2.  `t1 = a + t0`
    3.  `x = t1`
*   프로젝트 코드와의 연결: `10wk/minic_ir/src/codegen.c`의 `gen` 함수와 `new_temp()` 함수가 이 과정을 수행합니다.

### **4. 힌트 4: 코드 최적화 (이론) (상세 버전)**

*   핵심: 이 부분은 코드 파일이 아닌 **수업 이론 내용(PDF)**을 묻는 문제입니다. 아래 요약된 개념들을 중심으로 암기하세요.
*   코드 최적화란?: 프로그램의 의미(결과)를 바꾸지 않으면서, 실행 시간 단축, 메모리 사용량 감소 등을 목표로 코드를 변환하는 컴파일러의 핵심 기능입니다.

#### 최적화 기법의 분류

최적화는 적용되는 범위(Scope)와 시점에 따라 나눌 수 있습니다.

**1. 기계 독립적 최적화 (Machine-Independent Optimization)**

*   특징: 특정 CPU 구조와 관계없이, 중간 코드(IR) 단계에서 수행됩니다.
*   주요 기법:
    *   **지역 최적화 (Local Optimization)**: 제어 흐름의 변화가 없는 코드 블록인 **기본 블록(Basic Block)** 안에서만 수행됩니다.
        *   **공통 부분식 제거**: 블록 내에서 반복 계산되는 동일한 표현식을 한 번만 계산합니다. (`a=b+c; d=b+c;` → `t=b+c; a=t; d=t;`)
        *   **상수 폴딩**: 컴파일 시점에 계산 가능한 상수를 미리 계산합니다. (`a = 2 + 3;` → `a = 5;`)
        *   **죽은 코드 제거**: 실행 결과에 아무런 영향을 주지 않는 코드(사용되지 않는 변수에 값을 할당 등)를 제거합니다.
    *   **루프 최적화 (Loop Optimization)**: 프로그램 실행 시간의 90%가 루프에서 소요되므로 매우 중요합니다.
        *   **코드 이동 (Code Motion)**: 루프 반복에 따라 변하지 않는 **루프 불변식(loop-invariant)**을 루프 밖으로 꺼냅니다. (`for(...) a[i] = x*y;` → `t=x*y; for(...) a[i]=t;`)
        *   **세기 감축 (Strength Reduction)**: 비싼 연산을 싼 연산으로 대체합니다. (예: `y = x * 4` → `y = x << 2`)
    *   **전역 최적화 (Global Optimization)**: 기본 블록의 경계를 넘어 함수 전체에 걸쳐 최적화를 수행하며, **자료 흐름 분석(Data-Flow Analysis)**이 필요합니다.

**2. 기계 종속적 최적화 (Machine-Dependent Optimization)**

*   특징: 타겟 기계(CPU)의 구조를 고려하여, 주로 코드 생성 단계에서 수행됩니다.
*   주요 기법:
    *   **핍홀 최적화 (Peephole Optimization)**: 생성된 목적 코드의 인접한 명령어 몇 개를 보고 더 효율적인 명령어로 대체합니다.
        *   **효율적인 명령어 선택**: `x = x + 1`에 해당하는 명령어를 `INC x` 명령어로 대체.
        *   **중복 명령어 제거**: `STORE R1, x` 바로 다음에 `LOAD R1, x`가 오는 경우 `LOAD` 제거.
    *   **레지스터 할당**: 자주 사용되는 변수를 메모리 대신 빠른 레지스터에 할당합니다.
    *   **명령어 스케줄링**: CPU 파이프라인 특성을 고려하여 명령어 실행 순서를 재배치합니다.

### **5. 그 외 반드시 알아야 할 개념 (코드 생성 파트)**

*   함수 호출과 스택 프레임:
    *   함수 프롤로그 (함수 시작):
        1.  `pushl %ebp` : 이전 함수의 스택 프레임 기준점(`%ebp`)을 저장.
        2.  `movl %esp, %ebp`: 현재 스택 포인터(`%esp`)를 새로운 스택 프레임의 기준점으로 설정.
        3.  `subl $N, %esp` : 지역 변수를 저장할 공간 `N`바이트를 확보.
    *   함수 에필로그 (함수 끝):
        1.  `leave` : `movl %ebp, %esp` 와 `popl %ebp`를 합친 것. 스택 프레임을 정리.
        2.  `ret` : 함수 호출 지점으로 복귀.
*   변수 접근:
    *   지역 변수: `%ebp` 기준으로 음수 오프셋에 저장됨. (예: `-4(%ebp)`)
    *   매개변수(인자): `%ebp` 기준으로 양수 오프셋에 저장됨. (예: `8(%ebp)`)
*   C 호출 규약 (cdecl):
    *   인자 전달: 함수 호출자(caller)가 인자를 역순으로 스택에 넣는다.
    *   스택 정리: 함수 호출자(caller)가 호출이 끝난 후 스택을 정리한다.
    *   반환 값: `%eax` 레지스터를 통해 반환된다.
*   주요 레지스터 (퀴즈 힌트):
    *   `%eax`: 연산 결과 또는 함수 반환 값 저장.
    *   `%ebp`: 스택 프레임의 기준점(Base).
    *   `%esp`: 스택의 최상단(Top)을 가리킴.

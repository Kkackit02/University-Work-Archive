이 답안은 `compiler_study_guide.md`와 제공된 코드베이스의 분석을 기반으로 작성되었습니다.

## 1. 컴파일러 개요 및 기본 개념

1. 컴파일러의 주요 6단계(어휘 분석, 구문 분석, 의미 분석, 중간 코드 생성, 코드 최적화, 코드 생성)를 나열하고 각 단계의 입력과 출력을 설명하시오.

    *   어휘 분석 (Lexical Analysis):
        *   입력: 소스 코드 문자 스트림 (예: C 언어 코드)
        *   출력: 토큰 스트림 (예: `INT`, `IDENT("x")`, `ASSIGN`, `NUMBER(10)`, `SEMICOLON`)
        *   역할: 소스 코드를 의미 있는 최소 단위인 토큰으로 분리합니다.
    *   구문 분석 (Syntax Analysis):
        *   입력: 토큰 스트림
        *   출력: 추상 구문 트리 (Abstract Syntax Tree, AST)
        *   역할: 토큰 스트림이 언어의 문법 규칙을 따르는지 검사하고, 프로그램의 구조를 계층적인 트리 형태로 표현합니다.
    *   의미 분석 (Semantic Analysis):
        *   입력: 추상 구문 트리 (AST)
        *   출력: 의미적으로 유효함이 검증된 AST (주석이 달린 AST 또는 변경된 AST)
        *   역할: 프로그램의 의미적 오류(예: 타입 불일치, 미선언 변수 사용, 중복 선언)를 검사하고, 심볼 테이블을 구축하여 식별자 정보를 관리합니다.
    *   중간 코드 생성 (Intermediate Code Generation):
        *   입력: 의미 분석된 AST
        *   출력: 중간 코드 (Intermediate Representation, IR) (예: 3-주소 코드)
        *   역할: 타겟 머신에 독립적인 형태로 프로그램을 표현하여, 최적화 및 다양한 타겟으로의 코드 생성을 용이하게 합니다.
    *   코드 최적화 (Code Optimization):
        *   입력: 중간 코드 (IR)
        *   출력: 최적화된 중간 코드 (IR)
        *   역할: 프로그램의 성능을 향상시키기 위해 중간 코드를 개선합니다 (예: 불필요한 코드 제거, 레지스터 할당 최적화). (이 프로젝트에서는 크게 다루지 않음)
    *   코드 생성 (Code Generation):
        *   입력: (최적화된) 중간 코드 또는 AST
        *   출력: 타겟 머신 코드 (예: x86 어셈블리 코드)
        *   역할: 중간 코드를 실제 CPU가 이해하고 실행할 수 있는 타겟 머신(어셈블리 또는 기계어) 코드로 변환합니다.

2. 이 프로젝트에서 `compiler_class-main` 디렉토리와 `wk` 디렉토리들이 존재하는 방식이 컴파일러 학습에 어떤 이점을 제공한다고 생각하는가?

    이러한 디렉토리 구조는 컴파일러 학습에 매우 효과적입니다.
    *   점진적 학습: `wk` (week) 디렉토리들은 컴파일러의 각 단계를 주차별로 나누어 구현하는 과정을 보여줍니다. 예를 들어, 초기 주차에는 간단한 계산기만 파싱하거나 AST를 생성하는 데 집중하고, 이후 주차에는 심볼 테이블, 타입 시스템, IR, x86 코드 생성 등으로 기능을 확장하는 것을 단계적으로 학습할 수 있습니다.
    *   모듈별 이해: 각 디렉토리가 특정 단계나 기능에 초점을 맞추므로, 컴파일러의 각 모듈(렉서, 파서, AST, 시맨틱 분석기, 코드 생성기)의 역할을 명확히 이해하고 분리하여 학습할 수 있습니다.
    *   진화 과정 확인: 시간이 지남에 따라 컴파일러의 기능이 어떻게 추가되고 기존 코드가 어떻게 수정, 개선되는지 실제 코드를 통해 추적할 수 있습니다. 이는 복잡한 시스템이 개발되는 과정을 이해하는 데 도움이 됩니다.
    *   재사용 가능한 구성 요소: `compiler_class-main`은 아마도 기본 틀이나 초기 버전을 제공하고, 학생들은 이를 기반으로 각 `wk` 과제를 수행했을 것입니다. 이는 실용적인 프로젝트 개발 방식과 유사하며, 코드 재사용 및 확장의 개념을 자연스럽게 익힐 수 있습니다.

## 2. 어휘 분석 (Lexical Analysis - Flex)

1. `Assignment_My_Compiler/parser/lexer.l` 파일에서 `%option noyywrap`의 역할은 무엇이며, 이것이 컴파일러 동작에 어떤 영향을 미치는지 설명하시오.

    `%option noyywrap`는 Flex에게 입력 스트림의 끝(EOF)에 도달했을 때 `yywrap()` 함수를 호출하지 않도록 지시합니다.
    *   `yywrap()` 함수의 역할: 일반적으로 Flex는 입력 소스(파일)가 끝나면 `yywrap()` 함수를 호출하여 다음 입력 소스(예: 다음 파일)가 있는지 확인합니다. `yywrap()`이 0이 아닌 값을 반환하면 더 이상 입력이 없다고 판단하고, 0을 반환하면 새로운 입력 소스가 제공되었다고 간주하여 계속 스캔을 진행합니다.
    *   `noyywrap`의 영향: `noyywrap` 옵션이 설정되면 Flex는 `yywrap()`을 호출하는 대신, 입력 스트림의 끝에 도달하면 즉시 `yylex()` 함수가 0을 반환하도록 합니다. 이는 컴파일러가 단일 입력 파일만 처리하고 더 이상의 입력이 없을 때 유용하며, 불필요한 `yywrap()` 호출 오버헤드를 줄일 수 있습니다.

2. 다음 Flex 규칙의 동작을 설명하고, `yylval`이 어떻게 사용되는지 설명하시오.
    ```flex
    [0-9]+                  { yylval.int_value = atoi(yytext); return NUMBER; }
    [a-zA-Z_][a-zA-Z0-9_]*  { yylval.ident = strdup(yytext); return IDENT; }
    ```

    *   `[0-9]+ { yylval.int_value = atoi(yytext); return NUMBER; }`:
        *   동작: 하나 이상의 숫자로 구성된 문자열(예: "123", "0", "45")이 입력에서 매치되면 이 규칙이 실행됩니다. 매치된 문자열은 `yytext`에 저장됩니다.
        *   `yylval` 사용: `atoi(yytext)`를 통해 `yytext`에 저장된 문자열 "123"을 실제 정수 값 123으로 변환합니다. 이 정수 값을 Bison 파서로 전달하기 위해 `yylval.int_value`에 저장합니다. 그리고 Bison에 `NUMBER` 토큰 타입을 반환하여 토큰화가 완료되었음을 알립니다.
    *   `[a-zA-Z_][a-zA-Z0-9_]* { yylval.ident = strdup(yytext); return IDENT; }`:
        *   동작: 알파벳 문자(`a-zA-Z`)나 밑줄(`_`)로 시작하고, 이후에 알파벳 문자, 숫자(`0-9`), 밑줄이 0개 이상 이어지는 문자열(예: "myVar", "i", "variable_name")이 매치되면 이 규칙이 실행됩니다.
        *   `yylval` 사용: `yytext`에 저장된 매치된 식별자 문자열(예: "myVar")을 `strdup(yytext)`를 사용하여 동적으로 할당된 새로운 메모리 공간에 복사합니다. 이 복사된 문자열의 포인터를 `yylval.ident`에 저장하여 Bison 파서로 전달합니다. `strdup`를 사용하는 이유는 `yytext`가 내부 버퍼로, 다음 `yylex()` 호출 시 내용이 변경될 수 있기 때문에 값을 파서로 안전하게 전달하기 위함입니다. 이후 Bison에 `IDENT` 토큰 타입을 반환합니다.

3. Flex에서 정규 표현식 `[ 	
]+`와 `"//".*`가 각각 어떤 역할을 하며, 이러한 규칙이 컴파일러의 효율성에 어떤 기여를 하는가?

    *   `[ 	
]+`:
        *   역할: 하나 이상의 공백(space), 탭(tab), 캐리지 리턴(carriage return), 개행 문자(newline) 시퀀스를 매치합니다. 이 규칙에 대한 액션이 비어 있으므로(세미콜론 `;`만 있음), 매치된 공백 문자들은 단순히 무시됩니다.
        *   효율성 기여: 컴파일러의 렉서가 실제 언어의 문법적 의미를 가지는 토큰들(키워드, 식별자, 연산자 등)에 집중할 수 있도록 합니다. 공백 문자는 대부분의 프로그래밍 언어에서 토큰을 구분하는 역할을 할 뿐, 그 자체로는 의미가 없으므로 파싱 단계로 전달될 필요가 없습니다. 이를 통해 파서의 복잡성을 줄이고 효율성을 높입니다.
    *   `"//".*`:
        *   역할: `//` 문자열로 시작하고, 이후에 어떤 문자든 0개 이상 이어지는 시퀀스(`.`는 개행 문자를 제외한 모든 문자, `*`는 0회 이상 반복)를 매치합니다. 즉, 한 줄 주석을 매치합니다. 이 규칙 또한 액션이 비어 있으므로, 매치된 주석은 무시됩니다.
        *   효율성 기여: 주석은 프로그램의 가독성을 높이기 위한 것이며, 컴파일된 코드에는 영향을 주지 않습니다. 렉서 단계에서 주석을 제거함으로써, 컴파일러의 후반 단계(파서, 의미 분석기, 코드 생성기)가 주석으로 인해 복잡해지거나 불필요한 처리를 할 필요가 없게 되어 전체 컴파일 과정의 효율성을 높입니다.

4. 만약 `lexer.l` 파일에 알 수 없는 문자(`Unknown character`)를 처리하는 규칙이 없다면 어떤 문제가 발생할 수 있으며, 이를 어떻게 해결할 수 있는가?

    *   발생할 수 있는 문제:
        *   `lexer.l` 파일에 `.` (dot) 규칙이 없고, 입력 소스 코드에 어떤 Flex 규칙에도 매치되지 않는 문자(예: `@`, `#`, `$`, `~` 등 컴파일러 언어에서 정의되지 않은 특수 문자)가 나타난다면, Flex 렉서는 기본적으로 이 문자를 처리할 수 없어 오류가 발생하고, 일반적으로 프로그램이 비정상적으로 종료됩니다. 이는 사용자에게 불친절한 경험을 제공하고, 컴파일러가 견고하지 못함을 의미합니다.
        *   오류 메시지가 모호하여 사용자가 문제의 원인을 파악하기 어려울 수 있습니다.
    *   해결 방법:
        *   일반적인 규칙 추가: `lexer.l` 파일의 규칙 섹션 끝에 `. { fprintf(stderr, "Unknown character: %s\n", yytext); }`와 같은 일반적인 규칙을 추가합니다.
            *   이 규칙은 다른 어떤 규칙에도 매치되지 않는 모든 단일 문자(`. `의 의미)를 매치합니다.
            *   매치되면 `yytext`에 저장된 알 수 없는 문자를 표준 오류 스트림(`stderr`)에 출력하여 사용자에게 어떤 문자가 문제인지 명확히 알려줍니다.
            *   필요하다면 `exit(1)`과 같이 프로그램을 종료하거나, `yylval`에 오류 정보를 담아 파서로 전달하여 파서 단계에서 오류를 처리할 수도 있습니다.
            *   이를 통해 컴파일러는 예상치 못한 입력에 대해 우아하게 대처하고, 사용자에게 유용한 오류 메시지를 제공할 수 있게 됩니다.

## 3. 구문 분석 (Syntax Analysis - Bison)

1. `parser.y` 파일에서 `%union` 지시어의 필요성과, `yylval.int_value`와 `yylval.ident`처럼 멤버를 직접 참조하는 이유를 설명하시오.

    *   `%union` 지시어의 필요성:
        `%union`은 Flex 렉서가 반환하는 토큰의 의미 값(semantic value)과 Bison 파서의 문법 규칙 액션에서 사용될 비터미널 심볼의 의미 값의 타입을 정의하는 데 사용됩니다. 컴파일러는 단순히 토큰 타입만 주고받는 것이 아니라, 토큰이 나타내는 실제 값(예: 숫자 123, 식별자 "x")이나 파싱 결과로 생성된 AST 노드(예: `Expr`, `Stmt`)와 같은 더 복잡한 정보를 주고받아야 합니다. `union`은 이처럼 다양한 타입의 값을 하나의 심볼에 연결할 수 있도록 하여, Flex와 Bison 간의 인터페이스 및 파서 액션 내부의 값 전달을 유연하게 만듭니다.
    *   멤버를 직접 참조하는 이유:
        `yylval.int_value`나 `yylval.ident`와 같이 멤버를 직접 참조하는 것은 `yylval`이 `union` 타입으로 선언되어 있기 때문입니다. `union`은 메모리 공간을 공유하므로, 현재 어떤 타입의 값이 유효한지 명시적으로 알려주기 위해 해당 멤버를 통해 접근해야 합니다. 예를 들어, `NUMBER` 토큰이 반환될 때는 `yylval.int_value`에 정수 값이 들어있을 것이고, `IDENT` 토큰이 반환될 때는 `yylval.ident`에 문자열 포인터가 들어있을 것이라는 약속(`%token <int_value> NUMBER`, `%token <ident> IDENT`)이 있기 때문에 해당 멤버를 통해 값을 안전하게 가져올 수 있습니다.

2. Bison에서 다음 연산자 우선순위 및 결합성 정의가 문법 분석에 어떤 영향을 미치는지 설명하시오.
    ```bison
    %right '='
    %left EQ NE
    %left LT GT LE GE
    %left '+' '-'
    %left '*' '/'
    ```

    이 지시어들은 연산자의 우선순위와 결합성(associativity)을 명시하여 문법의 모호성을 해결하고 파서가 올바른 파스 트리를 생성하도록 돕습니다.
    *   우선순위:
        `%left`와 `%right` 지시어는 나열된 순서대로 우선순위가 낮아집니다. 즉, 가장 아래에 있는 연산자들(`*`, `/`)이 가장 높은 우선순위를 가지며, 가장 위에 있는 연산자(`=`)가 가장 낮은 우선순위를 가집니다. 같은 줄에 있는 연산자들은 동일한 우선순위를 가집니다.
        예: `a + b * c`는 `b * c`가 먼저 계산됩니다.
    *   결합성:
        *   `%left`: 좌측 결합성(left associative)을 가집니다. 같은 우선순위의 연산자가 연속될 때 왼쪽에서 오른쪽으로 연산이 진행됩니다.
            예: `a - b - c`는 `(a - b) - c`로 파싱됩니다. (`-`는 `%left`)
        *   `%right`: 우측 결합성(right associative)을 가집니다. 같은 우선순위의 연산자가 연속될 때 오른쪽에서 왼쪽으로 연산이 진행됩니다. `=` 연산자는 대표적인 우측 결합성 연산자입니다.
            예: `a = b = c`는 `a = (b = c)`로 파싱됩니다. (`=`는 `%right`)
        *   `%nonassoc` (여기에는 없지만): 비결합성(non-associative)을 가집니다. 같은 우선순위의 연산자가 연속될 때 문법 오류를 발생시킵니다 (예: `a < b < c`는 오류).

    영향: 이 정의들은 파싱 과정에서 Shift/Reduce 충돌이나 Reduce/Reduce 충돌이 발생할 수 있는 지점에서 Bison에게 어떤 규칙을 적용해야 할지 명확한 지침을 제공합니다. 이를 통해 파서는 개발자가 의도한 대로 연산 순서를 따르는 AST를 생성할 수 있습니다.

3. `parser.y` 파일에서 `expr : expr '+' expr { $$ = new_binop_expr(BIN_ADD, $1, $3); }`와 같은 규칙이 AST를 생성하는 원리를 설명하시오. 특히 `$$`, `$1`, `$3`의 의미를 명확히 하시오.

    이 Bison 규칙은 `+` 연산을 포함하는 표현식을 파싱하고, 그 결과를 AST 노드로 만드는 과정을 보여줍니다.
    *   `expr : expr '+' expr`: 이는 `expr` (좌측 피연산자)와 `'+'` (연산자) 그리고 `expr` (우측 피연산자)가 순서대로 나타나면, 이 전체를 하나의 `expr`로 간주하라는 문법 규칙입니다.
    *   `{ $$ = new_binop_expr(BIN_ADD, $1, $3); }`: 이 부분은 파서가 위 문법 규칙에 매치되었을 때 실행하는 '액션(Action)'입니다. 여기서는 `new_binop_expr`라는 함수를 호출하여 AST 노드를 생성하고 있습니다.
        *   `$$`: 현재 규칙의 왼쪽 심볼(여기서는 가장 바깥의 `expr`)의 의미 값을 나타냅니다. `$$`에 할당된 값은 이 `expr`이 파싱된 결과로, 상위 규칙에서 `expr`을 참조할 때 사용됩니다. 이 경우, `new_binop_expr` 함수가 반환하는 `Expr` (이항 연산자 AST 노드의 포인터)가 `$$`에 저장됩니다.
        *   `$1`: 규칙의 오른쪽에서 첫 번째 심볼(여기서는 `expr`)의 의미 값을 나타냅니다. 이는 `'+'` 왼쪽에 있는 `expr`이 파싱되어 생성된 AST 노드 포인터(`Expr`)입니다. `new_binop_expr` 함수의 `lhs` (left-hand side) 인자로 전달됩니다.
        *   `$3`: 규칙의 오른쪽에서 세 번째 심볼(여기서는 두 번째 `expr`)의 의미 값을 나타냅니다. 이는 `'+'` 오른쪽에 있는 `expr`이 파싱되어 생성된 AST 노드 포인터(`Expr`)입니다. `new_binop_expr` 함수의 `rhs` (right-hand side) 인자로 전달됩니다.
    *   AST 생성 원리: Bison 파서는 `expr '+' expr` 구조를 인식하면, 먼저 재귀적으로 `$1`과 `$3`에 해당하는 하위 `expr`들을 파싱하여 각각의 AST 노드를 얻어냅니다. 이 노드들이 준비되면, `BIN_ADD` (덧셈 연산자 타입)와 함께 이 두 하위 노드를 `new_binop_expr` 함수에 전달하여 새로운 이항 연산자 AST 노드를 생성합니다. 이 새로운 노드가 현재 `expr` 규칙의 결과(`$$`)가 되어 상위 규칙으로 전달되는 방식으로, 소스 코드 전체에 대한 계층적인 AST가 구축됩니다.

4. Bison 파서가 Shift-Reduce 충돌을 감지했을 때, 이를 해결하기 위한 일반적인 방법 두 가지를 제시하고 설명하시오.

    Shift-Reduce 충돌은 파서가 `Shift` (입력 심볼을 스택에 넣음)할지 `Reduce` (스택의 심볼들을 문법 규칙으로 축소)할지 결정할 수 없을 때 발생합니다.
    *   연산자 우선순위 및 결합성 명시 (`%left`, `%right`, `%nonassoc`):
        *   설명: 가장 일반적이고 효과적인 해결 방법입니다. Bison은 `%left`, `%right`, `%nonassoc` 지시어를 사용하여 정의된 터미널(연산자)의 우선순위와 결합성 정보를 파싱 테이블 생성에 활용합니다.
        *   예시: `expr: expr '+' expr | expr '*' expr`와 같이 `+`와 `*` 연산자가 모두 포함된 문법에서 `a + b * c`와 같은 입력이 있을 때, `b`를 읽고 난 후 `*`를 만났을 때 `b`를 포함하는 `expr`을 `expr '+' expr` 규칙으로 축소할지(`Reduce`) 아니면 `*`를 `Shift`할지 모호해집니다. 이때 `*`가 `+`보다 우선순위가 높다고 선언되어 있으면 Bison은 `*`를 `Shift`하도록 결정합니다. 마찬가지로 `a - b - c`에서 두 번째 `-`를 만났을 때, `-`가 좌측 결합성으로 선언되어 있으면 첫 `a - b`를 먼저 `Reduce`하도록 결정합니다.
    *   문법 규칙 재작성:
        *   설명: 문법 규칙 자체를 변경하여 모호성을 제거하는 방법입니다. 예를 들어, 연산자 우선순위가 다른 표현식들을 분리된 비터미널로 정의하는 것입니다 (`expr` -> `term` -> `factor` 등).
        *   예시:
            ```bison
            // 충돌이 발생할 수 있는 문법
            expr: expr '+' expr | expr '*' expr | NUMBER;

            // 재작성하여 우선순위를 명시한 문법
            expr: expr '+' term | term;
            term: term '*' factor | factor;
            factor: NUMBER | '(' expr ')';
            ```
            이처럼 `expr`, `term`, `factor`와 같은 비터미널들을 계층적으로 정의함으로써 `*` 연산자가 `+` 연산자보다 높은 우선순위를 가지도록 문법적으로 강제할 수 있습니다. 이는 Bison에게 명시적인 지시어 없이도 모호성을 해결하는 효과를 줍니다.

4. `yyerror(const char *s)` 함수의 역할은 무엇이며, 파싱 오류가 발생했을 때 이 함수가 어떻게 호출되는지 설명하시오.

    *   역할: `yyerror(const char *s)` 함수는 Bison 파서가 입력 스트림에서 구문 오류(syntax error)를 발견했을 때 호출되는 사용자 정의 함수입니다. 파서가 더 이상 유효한 문법 규칙을 매치할 수 없거나, 예상치 못한 토큰을 만났을 때 이 함수를 호출하여 오류를 보고합니다.
    *   호출 방식:
        1.  Flex 렉서(`yylex()`)가 Bison 파서(`yyparse()`)에게 토큰을 전달합니다.
        2.  Bison 파서는 전달받은 토큰들과 스택에 쌓인 심볼들을 기반으로 다음 상태를 결정하고 문법 규칙을 적용합니다.
        3.  만약 파서가 현재 상태에서 어떤 `Shift`나 `Reduce` 액션도 수행할 수 없고, `error` 토큰도 처리할 수 없는 상황에 놓이면, 이는 구문 오류 상황으로 판단합니다.
        4.  이 때 Bison은 `yyerror("syntax error")`와 같이 미리 정의된 오류 메시지 문자열(`"syntax error"` 또는 더 구체적인 메시지)을 인자로 하여 `yyerror` 함수를 호출합니다.
        5.  개발자는 `yyerror` 함수 내에서 오류 메시지를 출력하거나, 오류 카운터를 증가시키거나, 복구 전략(예: `yyerrok`, `yyclearin` 같은 Bison 매크로 사용)을 시도하는 등의 처리를 구현할 수 있습니다. 프로젝트의 `parser.y` 예시에서는 `fprintf(stderr, "parse error: %s\n", s);`를 통해 표준 오류에 메시지를 출력하고 있습니다.

## 4. 추상 구문 트리 (Abstract Syntax Tree - AST)

1. AST는 파스 트리와 어떻게 다르며, 컴파일러의 후반 단계에서 AST를 사용하는 주된 이점은 무엇인가?

    *   AST와 파스 트리의 차이점:
        *   파스 트리 (Parse Tree, 또는 구문 트리): 입력 소스 코드의 모든 토큰과 모든 문법 규칙 적용을 그대로 반영하는 트리입니다. 즉, 언어의 문법 구조를 완전하고 상세하게 표현합니다. 여기에는 불필요한 터미널 심볼(예: 괄호, 세미콜론), 비터미널 심볼, 단일 자식 노드만 가지는 비터미널 등이 포함됩니다.
        *   추상 구문 트리 (Abstract Syntax Tree, AST): 파스 트리에서 프로그램의 의미와 관련 없는 구문 세부 정보(예: 괄호, 타입 선언 키워드 `int`, 세미콜론 등)를 제거하고, 프로그램의 핵심적인 구조와 의미만을 추상적으로 표현한 트리입니다. 각 노드는 프로그램의 연산자, 식별자, 리터럴, 제어 흐름 구조 등을 나타냅니다.
    *   AST 사용의 주된 이점:
        *   간결성 및 효율성: 의미 없는 구문 정보가 제거되어 트리가 훨씬 간결하고 작아집니다. 이는 컴파일러의 후반 단계(의미 분석, 중간 코드 생성, 코드 최적화, 코드 생성)에서 트리를 순회하고 처리하는 데 필요한 시간과 메모리를 절약하여 효율성을 높입니다.
        *   의미 분석 용이성: 프로그램의 의미적 구조를 직접적으로 반영하므로, 타입 검사, 심볼 테이블 조회 등 의미 분석 작업을 수행하기에 훨씬 적합합니다.
        *   타겟 독립성: AST는 특정 프로그래밍 언어나 타겟 머신에 덜 의존적인 추상적인 표현이므로, 다양한 언어 기능이나 타겟 아키텍처를 지원하는 데 유연합니다.
        *   코드 생성 용이성: AST에서 직접 중간 코드나 타겟 코드를 생성하기가 파스 트리보다 훨씬 쉽습니다. 불필요한 노드를 건너뛰는 복잡한 로직 없이 핵심적인 연산에만 집중할 수 있습니다.

2. `Assignment_My_Compiler/include/ast.h` 파일의 `struct Expr`와 `struct Stmt`에서 `union`을 사용하는 이유는 무엇이며, `ExprKind`나 `StmtKind`와 같은 `enum` 타입과 함께 어떻게 활용되는지 설명하시오.

    *   `union`을 사용하는 이유:
        `union`은 여러 멤버들이 메모리 공간을 공유하는 C 언어의 특별한 구조체입니다. `struct Expr`와 `struct Stmt`에서 `union`을 사용하는 주된 이유는 메모리 효율성과 유연한 노드 표현을 위함입니다. AST의 노드는 종류에 따라 필요한 정보(데이터)가 다릅니다. 예를 들어, 정수 리터럴 노드는 `int` 값만 필요하지만, 이항 연산 노드는 연산자 종류와 두 개의 자식 `Expr`가 필요합니다. `union`을 사용하면 이 모든 다른 정보들을 하나의 노드 구조체 안에 포함시키되, 실제로 메모리에는 현재 노드 타입에 필요한 가장 큰 크기의 멤버만 할당되도록 하여 메모리 낭비를 줄일 수 있습니다.
    *   `enum` 타입과 함께 활용되는 방식:
        `ExprKind`나 `StmtKind`와 같은 `enum` 타입은 각 `struct Expr` 또는 `struct Stmt` 노드가 정확히 어떤 종류의 표현식/문장 노드인지를 나타내는 태그(tag) 역할을 합니다.
        *   각 AST 노드는 `kind` 필드(예: `Expr->kind`, `Stmt->kind`)에 `enum` 값을 저장합니다.
        *   컴파일러의 후반 단계(예: 의미 분석기, 코드 생성기)에서 AST를 순회할 때, 노드의 `kind` 값을 먼저 확인합니다.
        *   `kind` 값에 따라 `union` 내부의 어떤 멤버가 현재 유효한 데이터를 포함하고 있는지 판단하고, 해당 멤버에 안전하게 접근하여 필요한 정보를 추출하거나 처리합니다.
        *   예를 들어, `Expr->kind`가 `EXPR_INT`라면 `Expr->u.int_value`에 접근하고, `EXPR_BINOP`이라면 `Expr->u.binop.op`, `Expr->u.binop.lhs`, `Expr->u.binop.rhs`에 접근합니다.

3. `Assignment_My_Compiler/src/ast.c`에서 `new_int_expr`, `new_binop_expr`, `new_if_stmt`와 같은 함수들이 AST 노드를 생성하는 과정을 설명하시오. 특히 메모리 할당(예: `calloc`)과 문자열 복사(예: `strdup_s`)의 중요성을 강조하시오.

    이 함수들은 AST 노드를 동적으로 생성하고 초기화하는 팩토리 함수(factory function)입니다.

    *   `new_int_expr(int value)` (정수 리터럴 노드 생성):
        1.  메모리 할당: `Expr e = (Expr )calloc(1, sizeof(Expr));`를 통해 `Expr` 구조체 하나를 저장할 수 있는 메모리 공간을 동적으로 할당하고, 할당된 메모리를 모두 0으로 초기화합니다. `calloc`은 `malloc`과 달리 메모리를 0으로 초기화해주므로, 초기화되지 않은 포인터나 값으로 인한 잠재적 오류를 방지하는 데 유리합니다.
        2.  종류 설정: `e->kind = EXPR_INT;`와 같이 노드의 종류를 `EXPR_INT`로 설정합니다.
        3.  값 초기화: `e->u.int_value = value;`와 같이 `union`의 `int_value` 멤버에 주어진 정수 값을 저장합니다.
        4.  반환: 새로 생성된 `Expr` 노드의 포인터를 반환합니다.

    *   `new_binop_expr(BinOpKind op, Expr lhs, Expr rhs)` (이항 연산자 노드 생성):
        1.  메모리 할당: `Expr e = (Expr )calloc(1, sizeof(Expr));`를 통해 `Expr` 구조체 하나를 할당하고 0으로 초기화합니다.
        2.  종류 설정: `e->kind = EXPR_BINOP;`로 설정합니다.
        3.  값 초기화: `e->u.binop.op = op;`, `e->u.binop.lhs = lhs;`, `e->u.binop.rhs = rhs;`와 같이 `union`의 `binop` 멤버에 연산자 종류와 좌측/우측 피연산자 노드 포인터를 연결합니다.
        4.  반환: 새로 생성된 `Expr` 노드의 포인터를 반환합니다.

    *   `new_if_stmt(Expr cond, StmtList then_body, StmtList else_body)` (if 문 노드 생성):
        1.  메모리 할당: `Stmt s = (Stmt )calloc(1, sizeof(Stmt));`를 통해 `Stmt` 구조체 하나를 할당하고 0으로 초기화합니다.
        2.  종류 설정: `s->kind = STMT_IF;`로 설정합니다.
        3.  값 초기화: `s->u.if_stmt.cond = cond;`, `s->u.if_stmt.then_body = then_body;`, `s->u.if_stmt.else_body = else_body;`와 같이 `union`의 `if_stmt` 멤버에 조건식, then 블록, else 블록에 해당하는 AST 노드 포인터들을 연결합니다.
        4.  반환: 새로 생성된 `Stmt` 노드의 포인터를 반환합니다.

    *   메모리 할당(`calloc`)의 중요성:
        AST는 파싱 과정에서 동적으로 생성되는 트리 구조이므로, 각 노드는 힙(heap) 영역에 할당되어야 합니다. `calloc`을 사용하면 필요한 크기만큼 메모리를 할당하고 동시에 모든 비트를 0으로 초기화하므로, 초기화되지 않은 포인터나 값으로 인한 예상치 못한 동작(null 포인터 역참조 등)을 방지하는 데 도움을 줍니다. 이는 컴파일러의 안정성과 디버깅 용이성을 높입니다.

    *   문자열 복사(`strdup_s`)의 중요성:
        `new_var_expr(const char *name)`와 같은 함수에서 변수 이름(`name`)을 저장할 때, `e->u.var_name = strdup_s(name);`와 같이 원본 문자열을 복사하는 것은 매우 중요합니다. Flex의 `yytext` 버퍼나 Bison의 `$n`에서 전달되는 문자열 포인터는 일반적으로 임시 버퍼를 가리키며, 다음 렉서/파서 호출 시 내용이 덮어쓰여질 수 있습니다. 만약 원본 포인터만 저장한다면, AST 노드가 나중에 참조될 때 잘못된(이미 변경된) 문자열을 읽을 위험이 있습니다. `strdup_s`(또는 `strdup`)를 사용하여 새로운 메모리 공간에 문자열을 복사함으로써, AST 노드는 고유하고 안전한 문자열 데이터를 가지게 됩니다.

4. 다음 C 코드가 주어졌을 때, 해당 코드에 대한 AST의 일부를 간략하게 그려보거나 텍스트로 표현하시오.

## 5. 의미 분석 (Semantic Analysis)

1. 심볼 테이블의 주된 역할은 무엇이며, 이 프로젝트의 `symbol_table.c`에서 스코프(Scope)를 관리하기 위해 "가드 노드(guard node)"를 사용하는 방법을 설명하시오. `scope_push()`와 `scope_pop()`의 동작 원리를 포함하시오.

    *   심볼 테이블의 주된 역할:
        심볼 테이블은 컴파일러의 의미 분석 단계에서 사용되는 핵심 자료구조로, 소스 코드에 나타나는 모든 식별자(변수 이름, 함수 이름, 매개변수 이름 등)에 대한 정보를 저장하고 관리합니다. 이 정보에는 식별자의 이름, 종류(변수/함수/매개변수), 타입, 유효 범위(스코프), 메모리 위치, 인자 수(함수의 경우) 등이 포함됩니다. 심볼 테이블을 통해 컴파일러는 다음을 수행합니다.
        1.  선언 검사: 식별자가 사용되기 전에 선언되었는지 확인합니다.
        2.  중복 선언/정의 검사: 같은 스코프 내에서 식별자가 두 번 이상 선언되지 않았는지 확인합니다.
        3.  타입 검사: 식별자의 사용이 선언된 타입과 일치하는지 확인합니다.
        4.  스코프 관리: 식별자의 유효 범위에 따라 접근을 제어하고, 이름 충돌을 해결합니다.
    *   "가드 노드(guard node)"를 사용한 스코프 관리:
        `symbol_table.c`에서는 해시 테이블(체인 방식)을 기반으로 심볼 테이블을 구현하며, 스코프를 관리하기 위해 "가드 노드"라는 특별한 심볼 노드를 사용합니다. 가드 노드는 `name` 필드가 `NULL`인 `Sym` 구조체입니다.
        *   `scope_push()` 동작 원리:
            새로운 스코프(예: 함수 진입, `{...}` 블록 진입)에 진입할 때 `scope_push()`가 호출됩니다. 이 함수는 `cur_level` (현재 스코프 깊이)을 1 증가시킨 다음, 각 해시 버킷의 연결 리스트 맨 앞에 새로운 가드 노드를 삽입합니다. 이 가드 노드들은 현재 스코프가 시작되는 지점을 표시하며, 이후 현재 스코프 내에서 선언되는 심볼들은 이 가드 노드들 뒤에 추가됩니다.
        *   `scope_pop()` 동작 원리:
            현재 스코프를 벗어날 때 `scope_pop()`이 호출됩니다. 이 함수는 각 해시 버킷을 순회하면서 `name` 필드가 `NULL`이 아닌 심볼 노드(즉, 현재 스코프에 속한 심볼)들을 모두 해제(`free`)하고 제거합니다. 가드 노드를 만나면 해당 버킷에서의 현재 스코프 심볼 제거를 중단하고, 마지막으로 그 가드 노드 자체도 해제합니다. 이 과정을 통해 현재 스코프 내의 모든 심볼들이 심볼 테이블에서 제거되며, `cur_level`은 1 감소합니다. 이는 스코프가 스택처럼 동작하도록 하여, 적절한 유효 범위 관리를 가능하게 합니다.

2. `9wk/minic_type/sema.c` 파일에서 `check_expr()` 함수가 변수의 타입을 추론하고 미선언 변수 사용 오류를 감지하는 과정을 설명하시오. 관련 심볼 테이블 함수(예: `sym_get`)의 역할도 포함하시오.

    `sema.c`의 `check_expr()` 함수는 AST의 표현식 노드를 재귀적으로 순회하며 다음 과정을 통해 변수의 타입을 추론하고 미선언 변수 사용 오류를 감지합니다.

    *   변수 타입 추론:
        `check_expr(Node n)` 함수 내부에서 `n->kind`가 `NK_VAR` (변수 참조 노드)인 경우를 처리할 때, `Sym s = sym_get(n->op);`를 호출합니다. 여기서 `n->op`는 변수 이름을 나타냅니다. `sym_get()` 함수는 심볼 테이블에서 해당 변수 이름에 대한 심볼 정보를 찾아 반환합니다. 심볼 `s`가 성공적으로 찾아지면, `return s->tkind;`를 통해 해당 변수의 선언된 타입(`tkind`)을 반환하여 표현식의 타입으로 추론합니다.

    *   미선언 변수 사용 오류 감지:
        `sym_get(n->op)` 호출 결과 `s`가 `NULL`이면, 이는 해당 변수 이름(`n->op`)에 해당하는 심볼을 심볼 테이블에서 찾지 못했음을 의미합니다. 즉, 변수가 사용되기 전에 선언되지 않았다는 뜻입니다. 이 경우 `if(!s){ err(n->line, "use of undeclared identifier '%s'", n->op); return TY_ERROR; }`와 같이 `err()` 함수를 호출하여 "use of undeclared identifier" 오류를 보고하고, `TY_ERROR`를 반환하여 오류가 상위 표현식으로 전파되도록 합니다.

    *   `sym_get()` 함수의 역할:
        `sym_get()`은 심볼 테이블에서 주어진 이름의 심볼을 검색하는 함수입니다. 이 함수는 현재 스코프부터 시작하여 상위 스코프(이전 스코프)로 올라가면서 검색을 진행합니다. 가장 먼저 발견된 심볼을 반환하므로, 스코프 규칙에 따른 변수의 가시성(visibility)을 정확하게 반영합니다. `check_expr()`는 이 `sym_get()`의 반환값을 통해 변수의 존재 여부와 그 타입 정보를 얻게 됩니다.

3. 다음 코드에서 `binop_type` 함수(`sema.c` 참고)에 따라 `c` 변수의 최종 타입은 무엇이며, 그 이유는 무엇인가?
    ```c
    int a = 10;
    float b = 2.5;
    /* ... */
    TypeKind type_of_c = binop_type(a의_타입, b의_타입);
    ```

    `a`의 타입은 `TY_INT`이고, `b`의 타입은 `TY_FLOAT`입니다.
    `binop_type(TY_INT, TY_FLOAT)` 호출 시:
    `binop_type` 함수의 로직은 다음과 같습니다.
    ```c
    static TypeKind binop_type(TypeKind a, TypeKind b){
      if(a==TY_ERROR || b==TY_ERROR) return TY_ERROR;
      if(a==TY_CHAR) a=TY_INT;
      if(b==TY_CHAR) b=TY_INT;
      if(a==TY_FLOAT || b==TY_FLOAT) return TY_FLOAT; // 둘 중 하나라도 TY_FLOAT이면 결과는 TY_FLOAT
      return TY_INT; // 아니면 TY_INT
    }
    ```
    1.  `a`는 `TY_INT`, `b`는 `TY_FLOAT`이므로 `TY_ERROR` 조건에 해당하지 않습니다.
    2.  `TY_CHAR` 조건에도 해당하지 않습니다.
    3.  `if(a==TY_FLOAT || b==TY_FLOAT)` 조건에서 `b==TY_FLOAT`가 참이므로, 이 조건문이 실행됩니다.
    4.  따라서 함수는 `TY_FLOAT`를 반환합니다.

    결론: `c` 변수의 최종 타입은 `TY_FLOAT` 입니다.

    이유: `binop_type` 함수는 일반적인 C 언어의 타입 승격(Type Promotion) 규칙을 따르고 있습니다. 이항 연산에서 피연산자 중 하나라도 `float` 타입이면, 더 정밀한 `float` 타입으로 다른 피연산자를 자동 변환하여 연산을 수행하고 `float` 타입의 결과를 반환합니다. 이는 데이터 손실을 최소화하고 정확도를 유지하기 위함입니다.

4. 함수 정의 시 `handle_func_def` (sema.c 참고) 함수가 심볼 테이블을 어떻게 활용하여 함수의 중복 정의를 방지하고 매개변수 정보를 등록하는지 설명하시오.

    `sema.c`의 `handle_func_def` 함수는 다음과 같은 방식으로 심볼 테이블을 활용합니다.

    1.  함수 정보 등록 및 중복 정의 방지:
        *   함수 이름(`fname`), 반환 타입(`ret`), 인자 수(`arity`)와 같은 함수 정보를 추출합니다.
        *   `if(!sym_put(fname, SYM_FUNC, ret, arity))`를 호출하여 이 함수 정보를 심볼 테이블에 `SYM_FUNC` 종류로 등록을 시도합니다.
        *   `sym_put` 함수는 현재 스코프에 이미 같은 이름의 심볼이 존재하면 `false`를 반환하여 등록에 실패합니다. `handle_func_def`는 이 `false` 반환값을 통해 함수의 중복 정의를 감지하고 `err()` 함수를 호출하여 오류를 보고합니다. 성공적으로 등록되면 다음 단계로 진행합니다.

    2.  새 스코프 시작:
        *   함수 본문 내에서 선언된 매개변수와 지역 변수들은 함수 스코프 내에서만 유효해야 합니다. 이를 위해 `scope_push()`를 호출하여 새로운 스코프를 시작합니다. 이렇게 하면 함수 내부의 식별자들이 외부(전역) 식별자들과 이름 충돌을 일으키지 않게 됩니다.

    3.  매개변수 정보 등록:
        *   함수의 파라미터 리스트를 순회하며 각 파라미터의 이름(`p->op`)과 타입(`pt`)을 추출합니다.
        *   각 파라미터에 대해 `if(!sym_put(p->op, SYM_PARAM, pt, 0))`를 호출하여 이들을 심볼 테이블에 `SYM_PARAM` 종류로 등록합니다. 이 과정에서 파라미터 이름의 중복(예: `void func(int a, int a)`)도 감지하여 오류를 보고합니다.

    4.  스코프 종료:
        *   함수 본문에 대한 의미 분석(`visit_block`)이 완료되면 `scope_pop()`을 호출하여 함수 스코프를 종료합니다. 이 때 `scope_pop()`은 현재 스코프에 등록된 모든 매개변수와 지역 변수 심볼들을 심볼 테이블에서 제거합니다.

    이러한 과정을 통해 `handle_func_def`는 함수의 유효 범위와 중복 여부를 정확하게 관리하여 의미적으로 올바른 프로그램만 컴파일러의 다음 단계로 진행하도록 합니다.

5. 타입 검사에서 "Narrowing Conversion"은 무엇이며, `sema.c`의 `handle_decl` 또는 `check_expr` 함수에서 이를 어떻게 감지하고 처리하는지 설명하시오.

    *   Narrowing Conversion (축소 변환):
        Narrowing Conversion은 더 넓은(더 많은 값을 표현할 수 있는) 데이터 타입의 값을 더 좁은(더 적은 값을 표현할 수 있는) 데이터 타입으로 변환할 때 발생하는 상황을 의미합니다. 이 변환은 일반적으로 값의 손실이나 정확도 저하를 초래할 수 있습니다. 예를 들어, `float` 값을 `int` 타입으로 변환할 때 소수점 이하 부분이 잘려나가는 것이 대표적인 Narrowing Conversion입니다. 이는 잠재적인 프로그램 오류의 원인이 될 수 있으므로, 컴파일러는 이를 경고하거나 오류로 처리하는 것이 일반적입니다.

    *   `sema.c`에서 감지 및 처리:
        `sema.c`의 `handle_decl` (변수 선언 처리)과 `check_expr` (표현식 타입 확인) 함수에서 Narrowing Conversion을 감지하고 오류를 보고하는 로직이 있습니다.

        1.  `handle_decl` (변수 선언 및 초기화):
            ```c
            // handle_decl 함수 내
            if(idecl->nchild==1 && idecl->child[0]){ // 초기화 표현식이 있다면
              TypeKind rt = check_expr(idecl->child[0]); // 초기화 표현식의 타입 확인
              if(t==TY_INT && rt==TY_FLOAT) // 선언된 타입(t)이 int이고 초기화 값 타입(rt)이 float인 경우
                err(idecl->line, "narrowing init from float to int in '%s'", name); // 오류 보고
              // ...
            }
            ```
            `handle_decl` 함수는 `int x = 2.5;`와 같이 `int` 타입 변수를 `float` 리터럴로 초기화하는 경우를 감지합니다. 선언된 변수의 타입(`t`)이 `TY_INT`이고 초기화 값 표현식의 타입(`rt`)이 `TY_FLOAT`이면, `float`에서 `int`로의 Narrowing Conversion이 발생하므로 오류 메시지를 출력합니다.

        2.  `check_expr` (할당 표현식):
            ```c
            // check_expr 함수 내, NK_ASSIGN 케이스
            case NK_ASSIGN: {
              // ...
              TypeKind rt = check_expr(n->child[0]); // 할당될 값의 타입 확인
              if(s->tkind==TY_INT && rt==TY_FLOAT) // 좌변 변수 타입이 int이고 우변 값 타입이 float인 경우
                err(n->line, "narrowing assignment to '%s' (float→int)", n->op); // 오류 보고
              // ...
            }
            ```
            `check_expr` 함수는 `int_var = float_val;`와 같이 `int` 타입 변수에 `float` 값을 할당하는 경우를 감지합니다. 변수(`s->tkind`)가 `TY_INT`이고 할당될 값(`rt`)이 `TY_FLOAT`이면, Narrowing Conversion이므로 오류를 보고합니다.

    이러한 감지 로직을 통해 컴파일러는 잠재적인 데이터 손실이나 의도치 않은 동작을 미리 사용자에게 알려주어, 더욱 견고하고 정확한 프로그램을 작성하도록 돕습니다.

## 6. 중간 코드 생성 (Intermediate Code Generation - IR)

1. 중간 코드(IR)를 사용하는 주된 이유 두 가지를 설명하고, 이 프로젝트에서 사용된 3-주소 코드(Three-Address Code, TAC)의 일반적인 형태와 장점을 설명하시오.

    *   중간 코드(IR)를 사용하는 주된 이유:
        1.  기계 독립적인 최적화 가능: IR은 소스 언어에도, 타겟 머신에도 의존하지 않는 추상적인 코드 형태입니다. 따라서 IR 단계에서 수행되는 최적화(예: 불필요한 코드 제거, 공통 부분식 제거)는 특정 CPU 아키텍처에 얽매이지 않고 여러 타겟 머신에 적용될 수 있습니다. 이는 최적화 로직의 재사용성을 높여줍니다.
        2.  다양한 타겟 아키텍처 지원 용이: 프론트엔드(소스 코드 -> IR)와 백엔드(IR -> 타겟 코드)를 분리함으로써, 새로운 소스 언어를 지원하려면 새로운 프론트엔드만 개발하면 되고, 새로운 타겟 머신을 지원하려면 새로운 백엔드만 개발하면 됩니다. 이는 컴파일러 개발의 모듈성을 높이고 확장성을 제공합니다.
    *   3-주소 코드 (Three-Address Code, TAC)의 일반적인 형태와 장점:
        *   일반적인 형태: 3-주소 코드는 `result = operand1 operator operand2`와 같은 형태로 표현되는 명령어 시퀀스입니다. 각 명령어는 최대 세 개의 주소(피연산자 또는 결과 저장 위치)를 가집니다. 프로젝트의 `IR` 구조체는 `IROp op; char dst[32]; char src1[32]; char src2[32];`로 이를 반영합니다.
            *   예시: `t1 = x * 5`, `t2 = 10 + t1`, `x = t2`
            *   여기서 `t1`, `t2`는 임시 변수입니다.
        *   장점:
            1.  단순성: 각 명령어가 매우 단순하고 명확하여, 분석, 변환, 최적화하기가 용이합니다. 복잡한 표현식을 여러 개의 간단한 TAC 명령어로 분해할 수 있습니다.
            2.  최적화 용이성: 각 명령어가 고립적(independent)이므로, 데이터 흐름 분석(data-flow analysis)이나 제어 흐름 분석(control-flow analysis)을 수행하여 코드 최적화를 적용하기에 매우 적합합니다.
            3.  코드 생성 용이성: TAC는 실제 CPU의 어셈블리 명령어와 유사한 형태를 가지므로, TAC에서 타겟 머신 코드로 변환하기가 상대적으로 쉽습니다.

2. `10wk/minic_ir/include/ir.h`에 정의된 `IR` 구조체와 `IROp` 열거형이 3-주소 코드를 어떻게 표현하는지 설명하시오.

    `10wk/minic_ir/include/ir.h`의 `IR` 구조체와 `IROp` 열거형은 3-주소 코드를 다음과 같이 표현합니다.

    *   `IROp` 열거형:
        `IROp` (`IR_ADD`, `IR_SUB`, `IR_MUL`, `IR_DIV`, `IR_LOADI`, `IR_MOV`, `IR_PRINT` 등)은 중간 코드 명령어의 연산자(operator)를 정의합니다. 각 값은 특정 3-주소 연산을 나타냅니다. 예를 들어, `IR_ADD`는 덧셈 연산, `IR_LOADI`는 상수를 로드하는 연산을 의미합니다.
    *   `IR` 구조체:
        `IR` 구조체는 하나의 3-주소 코드 명령어(instruction)를 나타냅니다.
        ```c
        typedef struct IR {
            IROp op;            // 어떤 연산인지 (연산자)
            char dst[32];       // 대입 대상 / 결과가 저장될 위치 (result)
            char src1[32];      // 첫 번째 소스 피연산자 (operand1)
            char src2[32];      // 두 번째 소스 피연산자 (operand2, 이항 연산에만 사용)
            struct IR* next;    // IR 명령어 리스트를 위한 포인터
        } IR;
        ```
        *   `op`: `IROp` 열거형 값을 저장하여 이 명령어가 어떤 종류의 연산인지를 나타냅니다.
        *   `dst`: 연산 결과가 저장될 목적지 주소입니다. 이는 임시 변수(`t0`, `t1` 등)의 이름이거나 실제 변수 이름일 수 있습니다.
        *   `src1`: 첫 번째 소스 피연산자입니다. 이는 숫자 상수, 변수 이름, 다른 임시 변수 이름 등을 나타낼 수 있습니다.
        *   `src2`: 두 번째 소스 피연산자입니다. 이항 연산(`IR_ADD`, `IR_MUL` 등)에만 사용되며, 단항 연산이나 `IR_LOADI`, `IR_MOV` 등에서는 빈 문자열로 처리될 수 있습니다.
        *   `next`: `IR` 구조체들을 연결 리스트 형태로 구성하여 여러 개의 중간 코드 명령어 시퀀스를 표현하는 데 사용됩니다.

    표현 방식: 예를 들어, `x = y + 10`이라는 소스 코드는 다음과 같은 3-주소 코드로 변환될 수 있으며, 각각이 하나의 `IR` 구조체 인스턴스로 표현됩니다.
    *   `IR_LOADI, "t0", "10", ""` -> `t0 = 10`
    *   `IR_ADD, "t1", "y", "t0"` -> `t1 = y + t0`
    *   `IR_MOV, "x", "t1", ""` -> `x = t1`

3. `10wk/minic_ir/src/codegen.c`의 `gen` 함수에서 `new_temp()` 함수는 어떤 역할을 하며, 이 함수가 생성하는 임시 변수는 코드 생성 과정에서 어떻게 사용되는가?

    *   `new_temp()` 함수의 역할:
        `new_temp()` 함수는 중간 코드 생성 과정에서 필요한 새로운 임시 변수 이름을 생성하는 역할을 합니다. 예를 들어, `t0`, `t1`, `t2`와 같이 고유한 이름을 순차적으로 생성하여 반환합니다. 이 함수는 전역 카운터(`temp_id`)를 사용하여 매번 새로운 고유한 이름을 보장합니다.

    *   임시 변수의 코드 생성 과정 사용:
        `gen` 함수는 AST를 순회하며 표현식의 값을 계산하는 IR 명령어를 생성합니다. 복잡한 표현식(예: `a + b * c`)은 여러 개의 간단한 3-주소 코드로 분해됩니다. 이때 각 중간 계산 결과는 임시 변수에 저장됩니다.
        *   중간 계산 결과 저장: `gen` 함수는 이항 연산(예: `AST_ADD`, `AST_MUL`)을 처리할 때, 먼저 좌측과 우측 피연산자에 대해 재귀적으로 `gen`을 호출하여 IR을 생성합니다. 이 재귀 호출의 결과로, 각 피연산자의 값이 저장된 임시 변수 이름이 반환됩니다.
        *   현재 연산의 결과를 저장하기 위해 `char t = new_temp();`를 호출하여 새로운 임시 변수 `t`를 생성합니다.
        *   그리고 `emit(IR_ADD, t, left_temp, right_temp);`와 같이 이 임시 변수 `t`를 목적지(`dst`)로 하는 IR 명령어를 생성하여, `left_temp`와 `right_temp`에 저장된 값을 연산한 결과를 `t`에 저장하도록 지시합니다.
        *   마지막으로 `gen` 함수는 생성된 새로운 임시 변수 `t`의 이름을 반환하여, 상위 AST 노드에서 이 결과를 다음 연산의 피연산자로 사용할 수 있도록 합니다.

    예시: `a + b * c`의 IR 생성 과정에서 임시 변수 사용
    1.  `gen(b)` -> `y` (변수 이름 그대로 반환)
    2.  `gen(5)` -> `t0 = 5` (IR_LOADI), `t0` 반환
    3.  `gen(y * 5)` -> `t1 = y * t0` (IR_MUL), `t1` 반환
    4.  `gen(10)` -> `t2 = 10` (IR_LOADI), `t2` 반환
    5.  `gen(10 + (y * 5))` -> `t3 = t2 + t1` (IR_ADD), `t3` 반환
    6.  `gen(x = (결과))` -> `x = t3` (IR_MOV), `x` 반환 (또는 할당을 위한 별도 IR)

    이처럼 임시 변수는 각 부분 표현식의 결과값을 저장하여, 복잡한 표현식을 여러 단계로 나누어 처리하고 데이터 흐름을 명확하게 연결하는 데 핵심적인 역할을 합니다.

4. 다음 AST 표현식에 대해 `10wk/minic_ir/src/codegen.c`의 `gen` 함수가 생성할 수 있는 IR 시퀀스를 작성하시오. (가상의 `t0`, `t1` 등을 사용)
    `x = 10 + y * 5;` (여기서 `x`, `y`는 변수라고 가정)

    `gen` 함수는 표현식을 평가하여 그 결과를 임시 변수에 저장하고, 최종적으로 변수 `x`에 할당해야 합니다.

    1.  `y * 5` 평가:
        *   `gen(y)` -> `y` (변수 이름 그대로 반환)
        *   `gen(5)` -> `t0 = 5` (IR_LOADI), `t0` 반환
        *   `gen(y * 5)` -> `t1 = y * t0` (IR_MUL), `t1` 반환
    2.  `10 + (y * 5)` 평가:
        *   `gen(10)` -> `t2 = 10` (IR_LOADI), `t2` 반환
        *   `gen(y * 5)` 결과 -> `t1`
        *   `gen(10 + (y * 5))` -> `t3 = t2 + t1` (IR_ADD), `t3` 반환
    3.  `x = (10 + y * 5)` 할당:
        *   `gen(x = (결과))`에 해당하는 할당 IR (이 코드에서는 `IR_MOV`가 사용될 수 있음)

    따라서 생성될 IR 시퀀스는 다음과 같습니다:

    ```
    t0 = 5         // IR_LOADI, "t0", "5", ""
    t1 = y * t0    // IR_MUL,   "t1", "y", "t0"
    t2 = 10        // IR_LOADI, "t2", "10", ""
    t3 = t2 + t1   // IR_ADD,   "t3", "t2", "t1"
    x = t3         // IR_MOV,   "x", "t3", "" (또는 할당 연산을 위한 별도 IR)
    ```
    (Note: `Assignment_My_Compiler`의 `gen` 함수는 AST의 `EXPR_ASSIGN` 케이스를 직접 처리하지 않고, AST에서 `STMT_ASSIGN`나 `EXPR_ASSIGN` 노드에 대한 `emit` 호출이 별도로 필요할 수 있습니다. 위 예시는 `10wk/minic_ir/src/codegen.c`의 `gen` 함수 로직과 `ir.h`에 정의된 `IR_MOV`를 활용한 일반적인 IR 시퀀스입니다.)

## 7. 코드 생성 (Code Generation - x86 Assembly)

1. `Assignment_My_Compiler/src/codegen_x86.c`에서 함수 프롤로그(`pushl %ebp`, `movl %esp, %ebp`, `subl $size, %esp`)와 에필로그(`leave`, `ret`)의 역할은 무엇이며, 스택 프레임 설정 및 해제 과정을 설명하시오.

    함수 호출 시 스택 프레임(Stack Frame)은 함수의 지역 변수, 매개 변수, 이전 스택 프레임 정보 등을 저장하는 스택의 특정 영역입니다. 프롤로그와 에필로그는 이 스택 프레임을 설정하고 해제하는 표준적인 절차입니다.

    *   함수 프롤로그 (Function Prologue): 함수가 시작될 때 실행됩니다.
        1.  `pushl %ebp`: 이전(호출자) 스택 프레임의 베이스 포인터(`%ebp`)를 스택에 저장합니다. 이렇게 하면 함수가 종료된 후 호출자의 스택 프레임을 올바르게 복원할 수 있습니다.
        2.  `movl %esp, %ebp`: 현재 스택 포인터(`%esp`)의 값을 `%ebp`에 복사합니다. 이제 `%ebp`는 현재 함수의 스택 프레임 시작 지점(스택의 맨 위)을 가리키는 기준점이 됩니다. 지역 변수 및 매개 변수는 이 `%ebp`를 기준으로 오프셋을 통해 접근됩니다.
        3.  `subl $stack_size, %esp`: 지역 변수 및 기타 임시 데이터 저장을 위한 공간을 스택에 할당합니다. `stack_size`만큼 `%esp` 값을 감소시켜 스택을 아래로 확장합니다.
        *   역할: 새로운 스택 프레임을 설정하고, 호출자의 컨텍스트를 보호하며, 지역 변수 공간을 확보합니다.
    *   함수 에필로그 (Function Epilogue): 함수가 종료되기 직전에 실행됩니다.
        1.  `leave`: 이 명령어는 두 가지 작업을 수행하는 매크로입니다.
            *   `movl %ebp, %esp`: `%esp`를 `%ebp` 값으로 설정하여 현재 스택 프레임의 지역 변수 공간을 모두 해제합니다.
            *   `popl %ebp`: 스택에서 이전 함수의 `%ebp` 값을 꺼내 `%ebp` 레지스터에 복원합니다.
        2.  `ret`: 스택의 최상단에 저장된 반환 주소(`call` 명령어가 스택에 푸시한 주소)를 `%eip` 레지스터로 로드하여 함수를 호출한 곳으로 제어를 반환합니다.
        *   역할: 스택 프레임을 정리하고, 호출자의 컨텍스트를 복원하며, 함수 호출자에게 제어를 돌려줍니다.

2. x86 어셈블리에서 지역 변수와 함수 매개변수는 `ebp` 레지스터를 기준으로 어떻게 접근되는지 설명하시오. (`alloc_locals_and_params` 함수 참고)

    `codegen_x86.c`의 `alloc_locals_and_params` 함수는 각 함수에 대한 `Var` 구조체 배열을 생성하여 지역 변수와 매개 변수의 `ebp` 기준 오프셋을 계산하고 관리합니다.

    *   `%ebp` 레지스터의 역할: `%ebp` (Base Pointer)는 함수 스택 프레임 내에서 기준점 역할을 합니다. 함수의 프롤로그에서 `%ebp`는 스택 프레임의 시작 주소로 설정됩니다. 이 덕분에 스택이 `push`/`pop` 연산으로 인해 `%esp`가 변하더라도 `%ebp`는 스택 프레임 내에서 변하지 않는 고정된 기준점을 제공합니다.

    *   매개 변수 접근:
        *   함수가 호출될 때, `call` 명령 전에 호출자가 인자들을 스택에 푸시합니다.
        *   `%ebp`가 설정된 후, 매개 변수들은 `ebp`의 양의 오프셋(base + offset)으로 접근됩니다.
        *   일반적으로 `ebp+4`에는 이전 `%ebp`가, `ebp+8`에는 첫 번째 인자가, `ebp+12`에는 두 번째 인자가 위치합니다. (`alloc_locals_and_params`에서 `param_offset = 8`로 시작)
        *   예시: 첫 번째 매개 변수 `param1`는 `8(%ebp)`로 접근. (`movl 8(%ebp), %eax`)

    *   지역 변수 접근:
        *   함수 프롤로그에서 `subl $size, %esp`를 통해 확보된 지역 변수 공간은 `%ebp` 아래쪽(스택 주소가 더 낮은 쪽)에 위치합니다.
        *   따라서 지역 변수들은 `ebp`의 음의 오프셋(base - offset)으로 접근됩니다.
        *   일반적으로 `ebp-4`에는 첫 번째 지역 변수, `ebp-8`에는 두 번째 지역 변수가 위치합니다. (`alloc_locals_and_params`에서 `local_offset = -4`로 시작)
        *   예시: 첫 번째 지역 변수 `local_var1`는 `-4(%ebp)`로 접근. (`movl -4(%ebp), %eax`)

    이러한 `%ebp`를 기준으로 한 오프셋 방식은 컴파일러가 변수 이름을 추상화하고, 실제 메모리 위치를 효율적으로 관리하여 어셈블리 코드를 생성할 수 있게 합니다.

3. `codegen_x86.c`의 `gen_binop` 함수에서 이항 연산(`+`, `-`, `*`, `/`)과 비교 연산(`==`, `!=`, `<`, `>`, `<=`, `>=`)이 각각 x86 어셈블리 명령어로 어떻게 변환되는지 예시를 들어 설명하시오. 특히 비교 연산의 경우, 결과 값이 `eax`에 어떻게 설정되는지 명확히 하시오.

    `gen_binop` 함수는 이항 연산에 대해 다음과 같이 x86 어셈블리 코드를 생성합니다.

    *   일반적인 이항 연산(`+`, `-`, `*`, `/`):
        1.  우측 피연산자 평가: `gen_expr(e->u.binop.rhs, ...)`를 호출하여 우측 피연산자의 값을 계산하고 그 결과를 `%eax` 레지스터에 저장합니다.
        2.  `%eax` 스택에 푸시: `pushl %eax`를 통해 이 `%eax`의 값을 스택에 임시로 저장합니다. 이는 좌측 피연산자를 평가하는 동안 `%eax`가 덮어쓰여지는 것을 방지하기 위함입니다.
        3.  좌측 피연산자 평가: `gen_expr(e->u.binop.lhs, ...)`를 호출하여 좌측 피연산자의 값을 계산하고 그 결과를 `%eax` 레지스터에 저장합니다.
        4.  스택에서 `%ecx`로 팝: `popl %ecx`를 통해 스택에 저장했던 우측 피연산자의 값을 `%ecx` 레지스터로 가져옵니다. 이제 `%eax`에는 좌측 피연산자 값이, `%ecx`에는 우측 피연산자 값이 들어있습니다.
        5.  실제 연산 수행:
            *   덧셈 (`+`, `BIN_ADD`): `addl %ecx, %eax` (`%eax` 레지스터에 `%ecx` 레지스터 값을 더하고 결과를 `%eax`에 저장: `eax = eax + ecx`).
            *   뺄셈 (`-`, `BIN_SUB`): `subl %ecx, %eax` (`%eax` 레지스터에서 `%ecx` 레지스터 값을 빼고 결과를 `%eax`에 저장: `eax = eax - ecx`).
            *   곱셈 (`*`, `BIN_MUL`): `imull %ecx, %eax` (`%eax` 레지스터와 `%ecx` 레지스터 값을 곱하고 결과를 `%eax`에 저장: `eax = eax * ecx`).
            *   나눗셈 (`/`, `BIN_DIV`):
                *   `cdq`: `%eax` 레지스터의 부호를 확장하여 `%edx:%eax` 레지스터 쌍을 만듭니다. `idivl` 명령어는 `%edx:%eax`를 나눌 대상으로 사용합니다.
                *   `idivl %ecx`: `%edx:%eax`를 `%ecx` 값으로 나눕니다. 몫은 `%eax`에, 나머지는 `%edx`에 저장됩니다.
        *   예시 (a + b):
            ```assembly
                ; gen_expr(b)
                movl $b_val, %eax
                pushl %eax
                ; gen_expr(a)
                movl $a_val, %eax
                popl %ecx   ; ecx = b_val
                addl %ecx, %eax ; eax = a_val + b_val
            ```

    *   비교 연산(`==`, `!=`, `<`, `>`, `<=`, `>=`):
        1.  피연산자 평가 및 스택 처리: 일반적인 이항 연산과 동일하게 좌측/우측 피연산자를 `%eax`, `%ecx`에 로드합니다.
        2.  비교 수행: `cmpl %ecx, %eax` 명령어를 사용하여 두 피연산자를 비교합니다. 이 명령어는 `%eax - %ecx`를 계산하고 그 결과에 따라 플래그 레지스터(Flags Register)의 비트들을 설정하지만, 결과 자체를 `%eax`에 저장하지는 않습니다.
        3.  결과 값 (`%eax`에 0 또는 1 설정): 비교 결과(참/거짓)는 플래그 레지스터에 저장되며, 이를 `%eax`에 0(거짓) 또는 1(참)로 변환해야 합니다.
            *   `movl $0, %eax`: 먼저 `%eax`를 0으로 초기화합니다.
            *   `setcc %al` 명령어 사용: `setcc`는 `cmpl` 명령으로 설정된 플래그들을 기반으로 조건을 확인하고, 조건이 참이면 `%al` (eax의 하위 8비트)에 1을, 거짓이면 0을 설정합니다.
                *   `BIN_EQ` (`==`): `sete %al` (ZF=1이면 1, 아니면 0)
                *   `BIN_NE` (`!=`): `setne %al` (ZF=0이면 1, 아니면 0)
                *   `BIN_LT` (`<`): `setl %al` (SF!=OF이면 1, 아니면 0)
                *   `BIN_GT` (`>`): `setg %al` (SF=OF이고 ZF=0이면 1, 아니면 0)
                *   `BIN_LE` (`<=`): `setle %al` (SF!=OF 또는 ZF=1이면 1, 아니면 0)
                *   `BIN_GE` (`>=`): `setge %al` (SF=OF이면 1, 아니면 0)
            *   `%al`에 1 또는 0이 설정된 후, `%eax`의 상위 비트들은 여전히 0이므로 `%eax` 전체가 0 또는 1 값을 가지게 됩니다.
        *   예시 (a == b):
            ```assembly
                ; gen_expr(b)  -> eax = b_val
                pushl %eax     ; b_val 스택에 푸시
                ; gen_expr(a)  -> eax = a_val
                popl %ecx      ; ecx = b_val
                cmpl %ecx, %eax ; a_val - b_val (플래그 설정)
                movl $0, %eax   ; eax 초기화
                sete %al        ; eax = (a_val == b_val ? 1 : 0)
            ```

4. C 호출 규약(cdecl)에 따라 함수 `foo(a, b)`를 호출하는 `gen_call` 함수(codegen_x86.c)의 동작을 어셈블리 코드와 함께 설명하시오. 인자 전달 및 스택 정리는 누가 담당하는가?

    `codegen_x86.c`의 `gen_call` 함수는 C 호출 규약(cdecl)을 따르며 `foo(a, b)`와 같은 함수 호출에 대해 다음과 같이 어셈블리 코드를 생성합니다.

    *   `gen_call` 함수의 동작 및 어셈블리 코드:
        1.  인자들을 스택에 푸시 (역순):
            `gen_call` 내의 `push_args_reverse` 헬퍼 함수는 함수 호출의 인자 리스트(`ExprList`)를 재귀적으로 순회하며, **마지막 인자부터 먼저** (`b` -> `a` 순서) 스택에 푸시합니다.
            *   각 인자에 대해 `gen_expr`을 호출하여 인자 값을 계산하고 그 결과를 `%eax`에 저장합니다.
            *   `pushl %eax` 명령어를 사용하여 인자 값을 스택에 푸시합니다.
            *   어셈블리 예시 (`foo(a, b)`):
                ```assembly
                    ; gen_expr(b)  -> eax = b의 값
                    pushl %eax     ; b의 값을 스택에 푸시
                    ; gen_expr(a)  -> eax = a의 값
                    pushl %eax     ; a의 값을 스택에 푸시
                ```
                (이 시점에서 스택에는 `... | 반환 주소 | b의 값 | a의 값 | ...` 형태로 인자들이 쌓임)
        2.  함수 호출:
            `call _foo` 명령어를 사용하여 `_foo` 함수를 호출합니다.
            *   `call` 명령어는 현재 `%eip` (Instruction Pointer)의 다음 명령어 주소를 스택에 푸시한 후, `_foo` 함수의 시작 주소로 점프합니다.
            *   어셈블리 예시: 
                ```assembly
                    call _foo
                ```
        3.  스택 정리:
            함수 호출이 끝나고 `_foo` 함수에서 반환되면, `eax` 레지스터에는 `_foo` 함수의 반환 값이 들어있습니다. 호출자가 스택에 푸시했던 인자들을 제거하여 스택을 원래대로 되돌려야 합니다.
            *   `addl $8, %esp` (인자 2개 * 4바이트/인자 = 8바이트) 명령어를 사용하여 스택 포인터(`%esp`)를 인자들이 차지했던 공간만큼 증가시킵니다 (즉, 인자들을 스택에서 제거).
            *   어셈블리 예시: 
                ```assembly
                    addl $8, %esp ; 인자 2개 (4바이트 * 2 = 8바이트) 스택에서 제거
                ```

    *   인자 전달 및 스택 정리 담당:
        *   인자 전달: 호출자(caller)가 함수를 호출하기 전에 인자들을 스택에 푸시하여 전달합니다.
        *   스택 정리: 호출자(caller)가 함수 호출 후 `addl` 명령을 통해 스택에 푸시했던 인자들을 제거하여 스택을 정리합니다.
        *   반환 값: 함수 호출의 반환 값은 일반적으로 `%eax` 레지스터를 통해 전달됩니다.

    이러한 cdecl 호출 규약은 C/C++ 프로그램에서 가장 흔히 사용되는 호출 규약 중 하나입니다.

5. `codegen_x86.c`의 `gen_stmt` 함수에서 `if` 문(`STMT_IF`)과 `while` 문(`STMT_WHILE`)과 같은 제어 흐름 문장이 x86 어셈블리 레이블(`L_else_X`, `L_endloop_X`)과 조건부/무조건부 점프 명령(예: `je`, `jmp`)을 사용하여 어떻게 구현되는지 설명하시오.

    `gen_stmt` 함수는 `if` 문과 `while` 문을 어셈블리 레이블과 점프 명령어를 사용하여 구현합니다. 이 과정에서 고유한 레이블(`if_count`, `while_count`로 생성)을 사용하여 각 제어 흐름 구조가 서로 충돌하지 않도록 합니다.

    *   `if` 문 (`STMT_IF`) 구현:
        C 코드: `if (cond) { then_body; } else { else_body; }`
        1.  조건 평가: `gen_expr(s->u.if_stmt.cond, ...)`를 호출하여 `cond` 표현식을 평가합니다. 결과(0 또는 1)는 `%eax`에 저장됩니다.
        2.  조건부 점프: `cmpl $0, %eax` (eax 값을 0과 비교) 다음에 조건부 점프 명령어 `je` (Jump if Equal)를 사용합니다.
            *   만약 조건(`eax`)이 거짓(0)이면 `else_label` (else 블록 시작) 또는 `end_if_label` (else 블록이 없는 경우 if 문 끝)으로 점프합니다.
            *   어셈블리 예시: 
                ```assembly
                    ; gen_expr(cond) -> eax = cond_result
                    cmpl $0, %eax
                    je .L_else_0  ; 조건이 거짓이면 else 레이블로 점프
                ```
        3.  then 블록 코드 생성: `then_body`에 대한 `gen_stmt`를 재귀적으로 호출하여 코드를 생성합니다.
        4.  무조건 점프: `then_body` 실행이 끝나면 `jmp end_if_label` 명령어를 사용하여 `else_body`를 건너뛰고 `if` 문 끝으로 점프합니다.
            *   어셈블리 예시: 
                ```assembly
                    ; then_body 코드
                    jmp .L_endif_0 ; else를 건너뛰고 if 끝으로
                ```
        5.  else 블록 코드 생성 (선택적): `else_body`가 존재하면 `else_label:`을 정의하고 `else_body`에 대한 `gen_stmt`를 재귀적으로 호출하여 코드를 생성합니다.
            *   어셈블리 예시: 
                ```assembly
                    .L_else_0: ; else 레이블
                    ; else_body 코드
                ```
        6.  `if` 문 끝: `end_if_label:`을 정의하여 `if` 문의 종료 지점을 표시합니다.

    *   `while` 문 (`STMT_WHILE`) 구현:
        C 코드: `while (cond) { body; }`
        1.  루프 시작 레이블: `loop_label:`을 정의합니다. `while` 문은 이 레이블로 다시 점프하여 조건을 반복적으로 검사합니다.
            *   어셈블리 예시: 
                ```assembly
                    .L_loop_0: ; 루프 시작 레이블
                ```
        2.  조건 평가: `gen_expr(s->u.while_stmt.cond, ...)`를 호출하여 `cond` 표현식을 평가합니다. 결과는 `%eax`에 저장됩니다.
        3.  조건부 점프: `cmpl $0, %eax` 다음에 `je end_loop_label` 명령어를 사용합니다.
            *   조건(`eax`)이 거짓(0)이면 `end_loop_label` (루프 끝)로 점프하여 루프를 종료합니다.
            *   어셈블리 예시: 
                ```assembly
                    ; gen_expr(cond) -> eax = cond_result
                    cmpl $0, %eax
                    je .L_endloop_0 ; 조건이 거짓이면 루프 끝으로 점프
                ```
        4.  루프 본문 코드 생성: `body`에 대한 `gen_stmt`를 재귀적으로 호출하여 코드를 생성합니다.
        5.  무조건 점프: `body` 실행이 끝나면 `jmp loop_label` 명령어를 사용하여 루프의 시작 지점으로 다시 점프, 조건을 재검사하도록 합니다.
            *   어셈블리 예시: 
                ```assembly
                    ; body 코드
                    jmp .L_loop_0 ; 루프 시작으로 점프
                ```
        6.  루프 끝 레이블: `end_loop_label:`을 정의하여 `while` 문의 종료 지점을 표시합니다.

    이러한 방식으로 어셈블리 레이블과 조건부/무조건부 점프 명령을 조합하여 고수준 언어의 복잡한 제어 흐름 구조를 저수준 어셈블리로 정확하게 매핑할 수 있습니다.

## 8. 전체 컴파일러 통합 및 테스트

1. `Assignment_My_Compiler/src/main.c` 파일이 컴파일러의 각 단계(어휘 분석, 구문 분석, 코드 생성)를 어떻게 통합하여 소스 코드로부터 어셈블리 파일을 생성하는지 전체 흐름을 설명하시오.

    `Assignment_My_Compiler/src/main.c` 파일은 컴파일러의 드라이버 역할을 하며, `int main()` 함수 내에서 다음 흐름으로 각 단계를 통합하여 소스 코드로부터 어셈블리 파일을 생성합니다.

    1.  입력 파일 설정 (어휘 분석 준비):
        *   `argc`와 `argv`를 통해 명령줄 인자를 확인합니다.
        *   `argv[1]`에 소스 파일 경로가 있으면 `fopen()`으로 해당 파일을 읽기 모드로 엽니다.
        *   열린 파일의 `FILE` 포인터를 전역 변수 `yyin`에 할당합니다. `yyin`은 Flex 렉서(`yylex()`)가 소스 코드를 읽는 데 사용하는 입력 스트림입니다. 만약 인자가 없으면 기본적으로 표준 입력(`stdin`)을 사용합니다.
    2.  출력 파일 설정 (코드 생성 준비):
        *   `freopen("out.s", "w", stdout)` 함수를 호출하여 표준 출력 스트림(`stdout`)을 "out.s"라는 이름의 파일로 리다이렉션합니다.
        *   이 작업은 코드 생성 단계에서 `printf()` 함수를 통해 출력되는 모든 어셈블리 코드가 콘솔이 아닌 "out.s" 파일에 저장되도록 합니다.
    3.  파싱 시작 (어휘 분석 + 구문 분석):
        *   `yyparse()` 함수를 호출합니다. 이 함수는 Bison이 생성한 파서의 진입점입니다.
        *   `yyparse()`는 내부적으로 `yylex()` (Flex 렉서)를 반복적으로 호출하여 `yyin`으로부터 토큰을 하나씩 얻습니다.
        *   `yyparse()`는 이 토큰들을 정의된 문법 규칙(`parser.y`)에 따라 분석하고, 규칙의 액션(`{ ... }`) 내에서 `new_..._expr` 또는 `new_..._stmt`와 같은 함수들을 호출하여 AST(추상 구문 트리)를 구축합니다.
        *   파싱이 성공적으로 완료되면, 프로그램의 전체 AST는 전역 변수 `g_program` (`FunctionList`)에 저장됩니다. 파싱 중 문법 오류가 발생하면 `yyerror()`가 호출되고 `yyparse()`는 0이 아닌 값을 반환하여 `main` 함수에서 오류를 처리할 수 있게 합니다.
    4.  코드 생성:
        *   파싱이 성공하고 `g_program`에 유효한 AST가 있다면, `gen_x86_program(g_program)` 함수를 호출합니다.
        *   `gen_x86_program` 함수는 `g_program` (AST)을 순회하면서, 각 AST 노드(함수, 문장, 표현식 등)에 해당하는 x86 어셈블리 코드를 생성합니다.
        *   생성된 어셈블리 코드는 `printf()` 함수를 통해 출력되며, `freopen`에 의해 "out.s" 파일에 기록됩니다.
    5.  컴파일러 종료: 모든 단계가 성공적으로 완료되면 `main` 함수는 0을 반환하여 정상 종료를 알립니다.

2. `Assignment_My_Compiler/test/` 디렉토리의 역할은 무엇이며, 이 디렉토리에 있는 테스트 파일을 활용하여 컴파일러의 특정 기능(예: `for` 루프 처리)이 올바르게 구현되었는지 어떻게 확인할 수 있는가?

    *   `Assignment_My_Compiler/test/` 디렉토리의 역할:
        이 디렉토리는 컴파일러의 다양한 기능(예: 산술 연산, 변수 할당, 조건문, 반복문, 함수 호출 등)이 올바르게 구현되었는지 검증하기 위한 테스트 케이스(Test Cases) 소스 코드 파일들을 포함하고 있습니다. 각 `.c` 파일은 특정 언어 기능이나 시나리오를 테스트하도록 설계되어 있습니다.

    *   테스트 파일 활용 및 기능 검증 방법 (예: `for` 루프 처리):
        `for` 루프 처리가 올바르게 구현되었는지 확인하려면 `test/test_for.c`와 같은 파일을 활용할 수 있습니다.
        1.  컴파일러 실행: `test_for.c` 파일을 입력으로 사용하여 컴파일러를 실행합니다.
            *   예시: `your_compiler_executable test/test_for.c` (여기서 `your_compiler_executable`은 `main.c`를 컴파일하여 생성된 실행 파일)
        2.  어셈블리 코드 확인: 컴파일러는 "out.s" 파일에 `test_for.c`에 대한 x86 어셈블리 코드를 생성합니다. 이 `out.s` 파일을 텍스트 편집기로 열어 내용을 확인합니다.
            *   `for` 루프가 어떻게 어셈블리 레이블(`L_for_loop_X`, `L_for_endloop_X`, `L_for_increment_X`)과 조건부/무조건부 점프 명령(`jmp`, `je`, `cmpl`)으로 변환되었는지 `codegen_x86.c`의 `gen_for_stmt` 로직과 비교하여 분석합니다.
            *   초기화, 조건, 증감, 본문 부분이 예상대로 어셈블리 코드로 변환되었는지 검토합니다.
        3.  실행 및 결과 검증 (선택적):
            *   생성된 `out.s` 파일을 어셈블러(예: `nasm`, `as`)로 오브젝트 파일(`test_for.o`)로 변환합니다.
            *   링커(예: `ld`, `gcc`)를 사용하여 오브젝트 파일을 실행 파일(`a.out` 또는 `prog`)로 만듭니다. (필요 시 `support.c` 등 런타임 라이브러리 링크)
            *   생성된 실행 파일을 실행하고 그 출력이 기대하는 `for` 루프의 동작 결과와 일치하는지 확인합니다. 예를 들어, `for` 루프가 `printf`로 특정 값을 출력한다면, 실제 출력도 동일해야 합니다.
            *   이는 컴파일러가 코드를 단순히 어셈블리로 변환하는 것을 넘어, 의미적으로도 올바르게 동작하는지 최종적으로 검증하는 방법입니다.

    `test/` 디렉토리는 컴파일러 개발 과정에서 회귀 테스트(regression testing)를 수행하여, 코드 변경이 기존 기능에 영향을 주지 않았는지 지속적으로 확인하는 데도 매우 중요합니다.

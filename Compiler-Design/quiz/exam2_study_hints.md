## `exam2.txt` 시험 대비 공부용 힌트 문서

이 문서는 `quiz/exam2.txt` 파일의 내용을 기반으로 컴파일러 과목 시험을 위한 핵심 공부 힌트를 제공합니다.

---

### 1. 어휘 분석 (Lexical Analysis / Scanning)

*   **주요 역할**: 소스 코드의 문자 스트림을 읽어서 의미 있는 최소 단위인 '토큰(Token)'으로 분리합니다.
*   **토큰의 구성**: 일반적으로 토큰의 '타입(type)'과 '값(value)'으로 구성됩니다.
*   **줄 번호/열 정보의 중요성 (샘플 1 관련)**:
    *   **핵심**: `진단 메시지 (Diagnostic message)`
    *   **설명**: 컴파일러가 오류를 발견했을 때, 사용자가 소스 코드에서 해당 오류의 정확한 위치를 파악하고 수정할 수 있도록 정밀한 정보를 제공하기 위함입니다. 이는 사용자 친화적인 오류 보고에 필수적입니다.

### 2. 구문 분석 (Syntax Analysis / Parsing)

*   **주요 역할 (샘플 3 관련)**:
    *   어휘 분석 단계에서 생성된 토큰들의 나열이 프로그래밍 언어의 '문법 규칙(Grammar Rules)'에 맞는지 검사합니다.
    *   문법에 맞다면, 소스 코드의 구조를 계층적으로 표현하는 '트리 구조(Parse Tree 또는 Abstract Syntax Tree)'를 생성합니다.
    *   **오답 노트**: 단순히 토큰으로 쪼개는 것은 어휘 분석, 타입/스코프 정보 저장은 의미 분석, 중간 표현을 기계어로 변환하는 것은 코드 생성/최적화 단계의 역할입니다.

### 3. 의미 분석 (Semantic Analysis)

*   **주요 역할**: 구문 분석이 생성한 트리 구조를 바탕으로 프로그램의 '의미적 정확성'을 검사합니다.
    *   예: 변수가 선언되었는지, 타입이 일치하는지, 함수 호출 인자의 개수가 맞는지 등을 확인합니다.
*   **주요 도구**: '심볼 테이블(Symbol Table)'을 활용하여 변수, 함수 등의 정보(타입, 스코프 등)를 관리합니다.

### 4. 컴파일러 구조의 전반부 (Front-end)

*   **구성 (샘플 2 관련)**: 어휘 분석, 구문 분석, 의미 분석, 그리고 중간 코드 생성(Intermediate Code Generation) 단계를 통틀어 '프론트엔드(Front-end)'라고 부릅니다.
*   **역할**: 소스 언어에 의존적이며, 소스 코드를 중간 표현(IR)으로 변환하는 역할을 담당합니다.

### 5. 트리 구조의 이해: 파스 트리 vs. 추상 구문 트리 (AST)

*   **파스 트리 (Parse Tree / Concrete Syntax Tree)**:
    *   **특징**: 문법 규칙의 모든 세부 사항을 반영하여, 소스 코드의 토큰 및 비단말(non-terminal) 심볼을 포함하는 상세한 트리입니다.
    *   **예**: 괄호, 세미콜론 등 문법적으로 필요하지만 의미적으로 중요하지 않은 요소들도 포함합니다.
*   **추상 구문 트리 (Abstract Syntax Tree, AST) (샘플 4 관련)**:
    *   **특징**: 소스 코드에서 괄호, 쉼표 등 문법적으로 필요한 '불필요한 기호'를 대부분 생략하고, 프로그램의 '의미상 중요한 구조(연산, 피연산자, 문장)'만을 노드로 표현한 간결한 트리입니다.
    *   **용도**: 컴파일러의 이후 단계(의미 분석, 코드 생성, 최적화)에서 활용하기에 파스 트리보다 훨씬 효율적입니다.
    *   **핵심**: 추상 구문 트리는 프로그램의 의미를 효과적으로 나타내는 추상화된 표현입니다.

### 6. 심볼 테이블 (Symbol Table)

*   **역할**: 프로그램 내에서 사용되는 모든 식별자(변수, 함수, 타입 등)의 이름과 그에 대한 정보(타입, 스코프, 저장 위치 등)를 저장하고 관리하는 자료구조입니다. 의미 분석 단계에서 필수적으로 사용됩니다.

### 7. 중간 표현 (Intermediate Representation, IR)

*   **개념**: 소스 언어와 타겟 머신 언어 사이에 존재하는 추상적인 코드 형태입니다.
*   **장점**: IR을 사용하면 컴파일러의 재활용성이 높아지고, 최적화 단계에서 다양한 변환을 적용하기 용이합니다. (샘플 3, 4의 선택지에 포함되어 중요성 인지)

### 8. 파서 파일 (Parser Files) (힌트 3 관련)

*   **개념**: `Flex` (렉서 생성기)와 `Bison` (파서 생성기, Yacc의 GNU 버전)과 같은 도구를 사용하여 문법 정의(`*.y`) 및 어휘 규칙 정의(`*.l`) 파일로부터 자동으로 생성되는 소스 코드 파일(예: `parser.tab.c`, `lex.yy.c`)을 의미합니다.
*   **역할**: 이 파일들은 컴파일러의 어휘 분석기와 구문 분석기를 실제로 구현하는 코드입니다.

### 9. 레지스터 이름 (힌트 2 관련)

*   **중요성**: 코드 생성 단계(백엔드)에서 프로그램 변수나 중간 결과가 CPU 레지스터에 어떻게 할당되는지 이해하는 데 필요합니다.
*   **예시 (x86 아키텍처 기준)**:
    *   `EAX` (Accumulator Register): 산술 연산 결과, 함수 반환 값 등에 주로 사용됩니다.
    *   `EBX` (Base Register): 데이터 포인터나 오프셋 주소 지정에 사용됩니다.
    *   `ECX` (Counter Register): 루프 카운터 등에 사용됩니다.
    *   `EDX` (Data Register): 산술 연산의 확장이나 I/O 포인터 등에 사용됩니다.
    *   `ESP` (Stack Pointer): 스택의 최상단 주소를 가리킵니다.
    *   `EBP` (Base Pointer / Frame Pointer): 현재 스택 프레임의 베이스 주소를 가리키며, 지역 변수나 함수 인자에 접근하는 데 사용됩니다.
    *   `EIP` (Instruction Pointer): 다음에 실행될 명령어의 주소를 가리킵니다.
*   **학습**: 각 레지스터의 기본적인 역할과 용도를 이해하는 것이 중요합니다.
